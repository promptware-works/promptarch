# Feedback on Principles of APR-15-16-17

A short discussion in a separate session leads to the results summarized below.
NOTE that in that session I only posted the principles not the APRs.

## APR-015: Feedback on Principle

This is well above the bar of what most teams write down, and the core instincts are right: composed-not-accumulated, trust labels as a first-class primitive, provenance, per-class budgets, and a protected safety tier with logged degradation. I'd sign the spirit of it. But a principle this dense is load-bearing, so let me stress the seams — there are about five places where it's either under-specified or has a genuine hole.

**1. The headline invariant has no defined behavior when it can't be satisfied.** "Never silently evicts safety-critical content" is a hard invariant, but you haven't said what happens when the safety-critical class *alone* exceeds budget. A hard invariant needs a defined failure mode for the case where it's impossible to honor, or it silently becomes unenforceable and something gives anyway. The answer you want is *fail closed*: the run aborts or degrades to a safe refusal rather than being forced to evict. Say so explicitly, or the guarantee is aspirational.

**2. "Never evicts" doesn't close the compaction loophole.** If your reduction policy summarizes to save tokens, safety-critical content can be *lossily paraphrased away* without ever being "evicted." Same harm, different mechanism, not covered by the wording. The invariant should be: safety-critical content is neither evicted nor lossily transformed — it's carried verbatim or the run fails. Otherwise your protected tier leaks through the summarizer.

**3. Trust labels need enforced semantics and must be unforgeable — the principle currently only says segments are labeled.** A label with no stated downstream effect is decoration. The point of the label is that sub-threshold content is treated as *data, never as instructions* — assert that consequence, not just the labeling. And assume an adversary forges the scheme: if labels/delimiters live in-band as string markers, untrusted content (a fetched page, an email) can mimic them to escalate itself. Labels must be unforgeable from within segment content — per-run random sentinels, or better, structural separation via the API's own message boundaries rather than string delimiters.

**4. Reduction is per-class, and the log should be a composition manifest.** "A declared, audit-logged policy" (singular) hides the real structure. Reduction is class-specific — history compacts, retrieved docs re-rank and drop, tool outputs truncate head+tail, system/safety never — applied in a *declared order of sacrifice*. And "audit-logged" is satisfiable by `reduced: true`, which is useless for forensics. Specify that the log is a full manifest of the final composition: segment IDs, classes, trust labels, token counts, and which reduction fired. That's what makes a run reconstructable after an incident.

**5. Two classes the principle doesn't name.** First, *self-authored* segments — in an agent loop the model's own prior plans and scratchpad re-enter context, and they can carry laundered untrusted content (a tool result the model summarized). What trust label do those get? Second, the unit of composition: is "every run" every *model call* in the loop, or once per user turn? The strong version re-derives (or at least re-validates budgets and invariants) on every call, because overflow and trust drift accumulate iteration by iteration, not turn by turn.

One thing I'd *not* change: keep precedence and trust as the two axes, but note internally that declared precedence is about conflict resolution, while the model's actual positional/recency bias is a separate force — your declaration only holds if placement and explicit in-band authority signaling reinforce it, not ordering alone.

A tightened version incorporating the above:

> The context window is a composed artifact: every model call re-derives it from canonically-sourced, trust-labeled segments under a declared precedence order and per-class token budget. Trust labels carry enforced semantics — sub-threshold content is treated as data, never as instructions — and are unforgeable from within segment content. When inputs exceed budget, reduction follows a declared per-class policy applied in a declared order of sacrifice, recorded as a full composition manifest. Safety-critical content is a closed, enumerated class that is never silently evicted *or lossily transformed*; if it alone cannot fit, the run fails closed rather than degrading.

The one place I'd push you to decide before finalizing wording: is your protected class *only* guardrail/policy instructions, or does it also cover task-critical grounding facts? Those want different eviction rules, and collapsing them into one "safety-critical" tier is where this kind of principle usually starts leaking in practice.

## APR-016: Feedback on Principle

Same spine-is-right, seams-need-work verdict as before. The "recall re-enters at origin trust and never elevates" clause is the crown jewel — it's precisely the property most memory systems botch, and building the whole principle around it is correct. Tiering and scope-binding are the right secondary axes. But there are about five places where it leaks, and one is a real hole.

**1. The principle governs the read path; the vulnerability is the write path.** "Recall re-enters at origin trust" presumes origin trust was *correctly stamped at admission* and stored unforgeably. But the principle is silent on writes — who is allowed to write memory, and what trust gets stamped. That's the whole ballgame. A memory written from a tool result the model paraphrased is untrusted content, and if the write path stamps it by where it landed rather than where it came from, "origin trust" is already laundered before recall ever runs. You need an explicit admission clause: memory is stamped at write time with the trust floor of its inputs, and the write path itself is a governed, least-privilege operation — not something any tool result can perform freely.

**2. Derivation launders trust unless provenance is transitive.** This is the memory-side twin of the compaction loophole in your last principle. If M3 is summarized/synthesized from M1 (trusted) and M2 (untrusted), M3 must inherit the *floor* of its ancestors, not shed its untrusted lineage in the merge. State it as a monotonic rule: derived memory takes the least-trusted ancestor's label, and on merge of conflicting-trust sources you take the min, never the max. Without this, "never elevates" holds for direct recall but breaks the moment the agent reflects, summarizes, or aggregates — which it does constantly.

**3. "Scope is enforced" hides both the axes and the catastrophic failure.** Scope is carrying enormous unspecified weight. Name the axes — tenant/user, session/task, agent-or-tool identity, time — because the catastrophic one is cross-tenant recall (user A's memory surfacing for user B), and it deserves to be called out by name rather than folded into a generic word. Two properties the wording needs: enforcement happens *at the store/query boundary in code*, not by asking the model to be careful; and scope is *default-deny* — a query that doesn't declare its scope gets nothing, not everything.

**4. "Bounded and forgettable" bundles two things, and forgetting has a hard problem inside it.** Bounded (TTL, decay) and forgettable (erasure on request) are different mechanisms. The trap is that deleting a row doesn't undo influence: if a memory has been derived into other memories, graduated into an artifact, or shaped a fine-tune, erasing the origin leaves the influence intact. Forgetting must *cascade* — deleting M invalidates everything derived from M — or it's cosmetic. And there's a real tension with your own audit requirement: you need to erase the content while retaining the *fact of* the memory for forensics, which means tombstones, not true deletion of the record.

**5. Graduation is the one sanctioned elevation channel — say so, because it resolves an apparent contradiction.** "Durably shapes behavior graduates into a governed artifact" is under-triggered (what counts — one recall? repeated recall? the failure mode here is untrusted content recalled often enough to become de facto policy, i.e. *laundering by repetition*) and it seems to contradict "never elevates." It doesn't, and the resolution is the elegant part: trust never elevates *automatically*, and graduation is the *only* elevation path, gated by review. Make that explicit and the two clauses stop fighting — graduation becomes the human-in-the-loop promotion step, the single audited doorway through which anything earns behavior-shaping trust.

One more class to name, carried over from last time: **self-authored memory.** Agents write memories about themselves ("the user prefers X"). Those are model-authored from possibly-untrusted observation and must sit at the trust floor of their inputs, never at system trust — otherwise the reflection loop is a free elevation channel that bypasses graduation entirely.

The thing I would *not* touch: keep origin-trust-preservation as the spine, and keep the tier/scope/retention triad. That structure is sound.

A tightened version:

> Agent memory is tiered, trust-labeled, and scope-bound state. Admission is governed: memory is stamped at write time with the trust floor of its inputs, by a least-privilege write path, and derived or merged memory inherits its least-trusted ancestor's label. Recall re-enters content at its stored origin trust and never elevates it automatically. Scope — tenant, session, agent identity, time — is enforced default-deny at the query boundary in code. Retention is bounded (expiry, decay) and forgettable (cascading erasure that invalidates derived memories, via tombstones that preserve the audit fact without the content). Trust elevation occurs only through graduation: a governed, reviewed promotion that is the single audited channel by which memory earns behavior-shaping status.

The decision I'd force before you finalize: **is your memory store append-only?** If a stored memory can be edited in place, you've opened a silent-mutation laundering path (rewrite the content, keep the trusted stamp) and you've complicated cascading erasure. Append-only with versioning closes both, but it costs you storage and makes "update the user's stated preference" a more involved operation than it looks. That trade — mutability convenience vs. provenance integrity — is the one that tends to get decided implicitly by whoever writes the storage layer, and then silently governs everything above it.

## APR-017: Feedback on Principle

Third one, same treatment. The classification-by-blast-radius spine is right, and "no failure is silent" is the correct third leg. But this principle has a structural problem the other two didn't: **it describes what the harness decides, and says nothing about the fact that the model is sitting inside the loop, reading the failure, and re-planning around it.** That's where most of the holes cluster.

**1. The model routes around fail-closed unless the harness enforces it.** If a fail-closed decision is delivered to the model as a tool error string, the model doesn't experience it as a halt — it experiences it as a signal to try a different tool. Denied on `transfer_funds`, it reaches for `execute_script`. That's an *undeclared fallback*, invented at runtime by the fuzzy layer, which is exactly what your harness principle forbids. Fail closed has to terminate or gate the loop in code, not return a string the model is free to interpret. Corollary: after a fail-closed event, the harness must constrain the action space, not merely refuse one call.

**2. Fallbacks must inherit the constraints of the path they replace.** The classic and genuinely dangerous pattern: the validator/policy-check/retrieval times out, and the fallback path is *the same operation without it*. Degradation must reduce **capability, never guarantees**. A fallback that is less constrained than its primary is a privilege escalation dressed as resilience. Say it directly: no fallback may weaken a safety property; if the only available fallback would, that path fails closed.

**3. "Fail closed" is under-defined for side-effecting and ambiguous-outcome actions.** Halting is safe for a *read*. For a write, halting mid-sequence leaves partial state, and the ugly case is the ambiguous outcome — the POST timed out and you don't know whether it committed. Fail-closed presumes you know it failed. You need: deny *new* consequential actions, compensate/roll back in-flight ones, and treat unknown-outcome as *presumed-committed* for safety purposes (never blind-retry a non-idempotent action). Which surfaces the missing third handling mode entirely: **retry**. Retry is neither fail-closed nor degrade, it's the most common thing harnesses actually do, and it's ungoverned here.

**4. Bounded per-failure ≠ bounded per-run.** Each fallback is individually bounded, but degradations *compose*: retrieval degraded, then context truncated, then a weaker model, then a shortened tool schema — every step within its declared bound, the aggregate far outside spec. Boundedness needs to be a **run-level budget**, with degradation state that is sticky and monotone rather than re-evaluated fresh each call. And a floor: degradation must never reach the machinery that classifies failures or enforces guardrails, or the principle disassembles itself under load.

**5. "Never silent" doesn't reach the actual silent failure in promptware — confabulation.** Your invariant protects the log and the operator. But the archetypal silent failure is: retrieval returns nothing, the tool call errors, and the model produces fluent, confident, success-shaped prose over the hole. The failure was logged. The *output* was silent. Two things needed: degraded output must be **marked as degraded** at the artifact boundary so downstream consumers (including the next agent, and the user) don't treat it as full-fidelity; and "not silent" needs an audience decomposition — operator, user, and downstream consumer are three different disclosure obligations, and what you tell the model is separately a manipulation surface you should keep thin.

Two smaller ones worth folding in:

- **Escalate is not a terminal state.** It's a blocking wait on a channel that can itself fail, with a human who may not answer. It needs a declared timeout and a declared expiry behavior — which must be fail-closed, or "escalate" quietly becomes "wait, then proceed."
- **The residual class is inverted.** "Other paths may degrade" makes degrade the default for anything not classified. Unclassified or unrecognized failures are precisely the ones whose blast radius you don't know. Default-deny: unknown class fails closed. Classification is a design-time property of the tool/path in code, never a runtime judgment by the model.

What I'd keep untouched: handling determined by *what the failure blocks* rather than by what kind of error it is. That's the correct axis and most teams get it wrong by classifying on error type. I'd only sharpen "safety-critical or consequential" into its two real dimensions — **irreversibility and blast radius** — since that's what actually drives the decision, and add *detectability* as the tiebreaker (an irreversible failure you can't detect afterward should be treated as more severe than one you can).

Tightened:

> A failure's handling is determined at design time by what it blocks — its irreversibility, blast radius, and detectability — never by runtime model judgment. Safety-critical, irreversible, or consequential paths fail closed: new consequential actions are denied, in-flight ones are compensated or rolled back, ambiguous outcomes are presumed committed, and the harness constrains the action space so the model cannot re-plan around the halt. Other paths may degrade only via a declared fallback that is never less constrained than the path it replaces — degradation reduces capability, never guarantees — within a run-level degradation budget that may never reach the guardrail or classification machinery itself. Retries are governed as a distinct mode, permitted only for idempotent operations under declared limits. Escalation carries a declared timeout whose expiry fails closed. Unclassified failures fail closed. No failure or degradation is silent: each is logged for the operator, disclosed to the user, and marked at the artifact boundary so degraded output is never consumed as full-fidelity.

The decision to force before you finalize: **is degraded mode a property of a call or of the run?** If per-call, the agent silently re-enters full-fidelity mode on the next step, your composition manifest from Principle 1 stops telling the truth, and a memory written during degradation (Principle 2) gets stamped as if conditions were nominal. If sticky-for-the-run, you get honest provenance and a real budget — but you must then define how a run *recovers*, or every long-lived agent ratchets monotonically downward into uselessness and the only exit is a fresh run. Nobody decides this explicitly; it gets decided by whoever writes the retry wrapper, and then it governs everything above it.
